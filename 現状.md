> 本分析に基づき、設計文書 (spec.md v0.3, implementation-architecture.md v0.3) を更新済み。
> Phase 1 完了。MediaWiki 記法対応 (見出し・テーブル・脚注・引用・画像・カテゴリ・main リンク・箇条書き・定義リスト・水平線・ネストリスト・nowiki・REDIRECT・マジックワード・poem・syntaxhighlight) を実装済み。

## 1. アプリの全体像

GitHub Pages で公開する Wikipedia 風の静的 Wiki 基盤。
編集は Obsidian 等の外部エディタに委譲し、PR マージを契機に自動デプロイする構成。

### 現在の動作形態

- **SPA (Single Page Application)** として動作。ハッシュルーティング (`#!article/...`, `#!missing/...`, `#!disambiguation/...`) で画面遷移する
- 記事データは `content/` の Markdown 原稿を `npm run build:content` で事前に JS モジュール (`src/data/generated/content-data.js`) へ変換し、ブラウザで読み込む
- 記事ごとの個別 HTML は生成しない。全記事データを 1 つの JS に載せてクライアントで描画している

## 4. 設計上の課題

### 4.1 SPA 構成の限界

現在の構成は全記事データを 1 つの JS ファイルに格納し、クライアントで描画する SPA。

- **スケーラビリティ**: 記事数が増えると `content-data.js` が肥大化し、初回ロードが重くなる
- **SEO**: ハッシュルーティングはクローラーに不利。OGP メタタグも記事ごとに設定できない
- **直リンク**: `#!article/...` 形式の URL は共有時に不安定になりやすい
- 仕様では「原稿から生成された HTML を正とする」(spec 8.8) と明記されており、最終的には記事ごとの HTML 生成に移行する必要がある

### 4.3 ~~記事本文のインライン書式~~ → 解決済み

インライン書式 (太字・斜体・コード・MediaWiki書式・外部リンク) はセクション本文で保持され、描画層で HTML に変換される。リストも `<ul>/<li>` として描画済み。画像埋め込み (`![[...]]`) と callout (`> [!note]`) も対応済み。MediaWiki テーブル・見出し・脚注・引用・カテゴリ・画像変換・main リンクもすべて実装完了。

### 4.4 標準 Markdown 記法の未対応部分

content-import.js のパーサーは MediaWiki 記法を主軸にしており、以下の標準 Markdown 記法には未対応。記事執筆時にはこれらの代替記法を使う必要がある。

| 未対応の Markdown 記法 | 代替手段 |
| ------ | ------ |
| フェンスドコードブロック (`` ``` ``) | `<syntaxhighlight>` / `<source>` を使用 |
| Markdown テーブル (`col1` `col2` をパイプ区切り) | MediaWiki テーブル (`{\| class="wikitable" ... \|}`) を使用 |
| 素の引用 (`> テキスト`) | `> [!type]` callout 記法、または `<blockquote>` を使用 |
| 水平線 `---` (3ダッシュ) | `----` (4ダッシュ以上) を使用。3ダッシュは先頭では frontmatter 区切り、本文中では水平線にならない |
| Markdown アンカーリンク (`[text](#anchor)`) | 同一ページ内の見出しジャンプは未対応。外部リンクとして処理される |

これらは記法対応の優先度と仕様方針（MediaWiki 記法を基礎とする）に基づき、現時点では意図的に未実装としている。

### 4.5 build:site が単なるファイルコピー

`build-site.mjs` は `index.html` と `src/` を `dist/` にコピーするだけ。バンドル、ミニファイ、HTML 生成は一切行わない。GitHub Pages に載るのは開発時と同じファイル構成。
