# 投稿経路とレビュー運用

## 1. この文書の目的

WikiLikePagesにおける投稿経路とレビュー運用の詳細をまとめる。
参加者の技術的背景に応じた複数の入力経路を、1つの運用方針として整理する。

- GitHubアカウントがある参加者
- GitHubアカウントはあるがGitは使いたくない参加者
- GitHubアカウントがない参加者
- OneDriveやGoogle Driveの共有ドライブ経由で原稿を受けたい場合

仕様の全体像は [spec.md](./spec.md) を、実装の詳細は [implementation-architecture.md](./implementation-architecture.md) を参照。

## 2. 基本原則

- 公開コンテンツの正本はGitHubリポジトリに置く
- 公開反映は必ずPull Requestと管理者マージを経由する
- GitHub以外の入力経路は、正本ではなく投稿箱として扱う
- 自動化は「PR作成まで」を基本にし、自動マージは原則採用しない

## 3. 投稿経路の全体像

WikiLikePagesでは、参加者の習熟度に応じて複数の投稿経路を持つ。

| レーン | 想定ユーザー | 入力場所 | GitHub側の着地 | 推奨度 |
| --- | --- | --- | --- | --- |
| PR投稿 | GitHubとGitが使える人 | ローカル編集 + Git | Pull Request | 最優先 |
| Issue Form投稿 | GitHubアカウントはあるがGitは避けたい人 | GitHub Issue Form | 管理者がPR化 | 推奨 |
| 外部フォーム投稿 | GitHubアカウントがない人 | 外部フォームやメール | 管理者がPR化 | 推奨 |
| 共有ドライブ投稿 | ドライブ共有に慣れている人 | OneDrive / Google Drive | Botまたは管理者がPR化 | 条件付き推奨 |

## 4. 標準レーン: Pull Request投稿

### 4.1 フロー

1. 執筆者がリポジトリをローカルにクローンする
2. ObsidianでVaultとして開く
3. ページを追加・編集する
4. GitHub DesktopまたはGitでコミットする
5. Pull Requestを作成する
6. 編集者または管理者がレビューする
7. 管理者が`main`へマージする
8. マージ済みのMarkdown原稿から公開用ページを自動生成する
9. GitHub ActionsがGitHub Pagesへデプロイする

### 4.2 このレーンの利点

- 差分が最も見やすい
- レビューの根拠が残る
- リンク修正やfrontmatter修正が執筆者側で完結しやすい
- 原稿と公開物を分けたまま運用できる

### 4.3 原稿と公開物の関係

- Pull Requestでレビューする対象はMarkdown原稿である
- 公開サイトで読者が見るのは、その原稿から生成された公開ページである
- 生成後のHTMLは成果物であり、レビューや修正の起点にはしない

## 5. 簡易レーン: GitHub Issue Form投稿

### 5.1 フロー

1. 投稿者がIssue Formを開く
2. タイトル、本文、タグ、関連ページ、添付を入力する
3. 管理者が内容を整形し、Markdownファイルへ変換する
4. 管理者が変換結果をPull Requestとして作成する
5. レビュー後にマージする

### 5.2 向いているケース

- GitHubアカウントは作れる
- しかしGit操作は避けたい
- 軽い追記や新規記事案を投げたい

## 6. 非GitHubレーン: 外部フォームまたはメール投稿

### 6.1 フロー

1. 投稿者が公開サイト上の投稿フォームまたは専用メールへ原稿を送る
2. 管理者が内容を確認する
3. 管理者がMarkdownへ変換し、Pull Requestを作成する
4. レビュー後にマージする

### 6.2 実務上の扱い

- 外部フォームはGoogle FormsやMicrosoft Formsのような既製サービスで十分
- 最初は手動取り込みでよい
- 投稿量が増えてから自動PR化を検討する

## 7. 共有ドライブレーン

### 7.1 結論

OneDriveやGoogle Driveの共有ドライブを使う案は成立する。
ただし、共有ドライブを正本にするのではなく、投稿箱として使うべき。

### 7.2 推奨フロー

1. 投稿者が共有ドライブへMarkdownや画像を置く
2. 取り込み用Botまたは管理者が差分を検出する
3. frontmatter補完や配置整理を行う
4. GitHub上にPull Requestを作成する
5. 管理者レビュー後にマージする

### 7.3 共有ドライブを正本にしない理由

- 変更履歴がGitほど読みやすくない
- 同時編集時の競合が見えにくい
- リネームや移動で内部リンクが壊れやすい
- frontmatterや記法の検証を通しにくい
- 誰の意図で修正されたかが追いにくい

### 7.4 自動マージを勧めない理由

- 誤配置や記法崩れをそのまま公開しやすい
- 添付ファイルの差し替えミスが起きやすい
- スパムや誤投稿を防ぎにくい
- 競合解決の判断を機械に寄せすぎる

### 7.5 共有ドライブ連携をやるなら必要な処理

- 変更ファイルの検出
- 取り込み対象フォルダの制限
- frontmatterの補完
- 禁止文字やリンクのチェック
- 画像や添付の配置変換
- PRタイトルと説明文への元データ情報の付与

## 8. 運用判断

### 8.1 最小構成

最初に採るならこの構成が最も軽い。

- 正本: GitHub
- 標準投稿: PR
- 非GitHub投稿: 外部フォームまたはメール
- 共有ドライブ: まだ使わない

### 8.2 投稿量が多い場合

投稿量が増えたら次を追加する。

- GitHub Issue Form
- 共有ドライブからのPR作成Bot

### 8.3 避けるべき構成

初期段階では次は避ける。

- 共有ドライブを正本にする
- 共有ドライブから直接`main`へ自動反映する
- 外部フォームからレビューなしで公開する

## 9. 未決事項

- 非GitHub投稿の主経路はメールかフォームか
- 共有ドライブを使うならOneDriveとGoogle Driveのどちらか
- 共有ドライブ連携は手動取り込みかBot化か
- 自動化をPR作成までで止めるか、一部自動マージを許すか

## 10. 推奨判断

現時点では、以下を推奨する。

- GitHubを正本にする
- 公開反映は管理者マージ必須にする
- GitHubアカウントなし参加者は外部フォームか共有ドライブへ投稿する
- 共有ドライブを使う場合でも、Botが作るのはPull Requestまでに止める
