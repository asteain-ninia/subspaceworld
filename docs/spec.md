# WikiLikePages 仕様 v0.3

## 1. 背景

- 共同創作に使うWikiサービスの選定が難航している。
- 参加者はWiki文化に慣れているため、ページ単位編集・内部リンク・タグ・注釈のような体験は維持したい。
- ただし運用コストと実装コストを抑えたいので、公開は静的ホスティングに寄せたい。
- そのため、公開基盤はGitHub Pages、編集体験は外部エディタに切り出す方向を第一候補とする。

## 2. 解きたい課題

- 読者向けには通常のWikiのように閲覧できること。
- 執筆者向けには、Wikiっぽい書き心地をなるべく崩さないこと。
- サーバー運用やDB運用を持たずに継続できること。
- 将来の拡張で、スタイルやテンプレートを増やせること。

## 3. MVPの目標

- MediaWiki風のWiki記法を基礎としつつ、標準MarkdownとObsidian記法も取り込める形でページを作成・編集できる。
- `[[ページ名]]` 形式の内部リンクが使える。
- タグ、目次、検索、バックリンクなどのWiki系ナビゲーションを提供する。
- 管理者レビューとマージを通った内容だけを公開する。
- GitHubへのマージ後、自動でGitHub Pagesへ反映される。
- 下書きページを公開対象から除外できる。
- GitHubアカウントがない参加者にも、記事データを送る最低限の窓口を用意する。

## 4. MVPでやらないこと

- GitHub Pages上で完結するブラウザ内WYSIWYG編集。
- 同時編集やリアルタイム共同編集。
- 細かい権限管理を持つ独自管理画面。
- 既存Wiki記法の完全互換。

## 5. 制約と前提

- GitHub Pagesは静的サイト配信が前提で、任意の静的サイトジェネレータはGitHub Actionsのカスタムワークフローで配信できる。
- GitHub Docsでは、GitHub PagesはGitHub Freeでは公開リポジトリで利用でき、非公開リポジトリでの利用はPro/Team/Enterprise系プランが前提とされている。
- Pages単体では安全な認証付き編集UIを持ち込みにくいため、ブラウザ内編集を先に作るより、編集をObsidianなどに委譲したほうがMVPの難易度が低い。

## 6. アーキテクチャ

### 6.1 採用構成

- 編集: ローカルでObsidianまたは任意のテキストエディタを使う
- ソース形式: MediaWiki風のWiki記法を基礎とし、標準MarkdownとObsidian記法を互換入力として扱う
- 投稿経路: GitHub Pull Requestを標準とし、簡易投稿経路を別途用意する
- 静的サイト生成: 自前の Node.js ビルドスクリプト (`scripts/`)
- 公開: GitHub Pages (GitHub Actions 経由でデプロイ)

投稿経路の詳細は [contribution-workflows.md](./contribution-workflows.md) を参照。
実装の詳細は [implementation-architecture.md](./implementation-architecture.md) を参照。

### 6.2 Quartz 4 を採用しなかった経緯

初期検討では Quartz 4 を第一候補とした。Quartz 4 は Obsidian 互換、全文検索、グラフ表示、wikilinks、transclusions、backlinks を公式機能として掲げており、要件との親和性が高かった。

しかし、既存の創作原稿に MediaWiki 風テンプレート (`{{基礎情報 国|...}}`) が多く含まれており、これを取り込むには Quartz のパーサを拡張するより、独自の正規化層を持つほうが制御しやすいと判断した。結果として、Node.js ベースの自前ビルドパイプラインを採用し、MediaWiki 風記法・Markdown・Obsidian 記法の混在を正規化層で吸収する構成にした。

### 6.3 採用理由

- Obsidianは公式に `[[Wikilinks]]` と通常Markdownリンクの両方をサポートし、ファイル名変更時に内部リンクを自動更新できる。
- Obsidianのcallout記法は公式にサポートされており、スタイル付き注釈を比較的低コストで書ける。
- 既存の創作原稿にはMediaWiki風の記法が含まれるため、そこを取り込み可能にしたほうが移行コストが低い。
- GitHub Pagesは公式にカスタムワークフロー経由で任意の静的サイトジェネレータを配信できる。
- GitHub Issue Formsはテキスト入力、ドロップダウン、チェックボックス、ファイルアップロードを持てるため、Gitを触らない投稿窓口として使いやすい。
- 外部フレームワークに依存せず自前でビルドすることで、MediaWiki風テンプレートの変換やリンク解決を自由に制御できる。

## 8. コンテンツ仕様

### 8.1 基本単位

- 1ページ = 1 テキスト原稿ファイル。対応拡張子は `.md`, `.wiki`, `.txt`
- 原稿置き場は `content/` とし、ページ本文だけでなく画像や添付ファイルもここに含める
- ページ間リンクは `[[ページ名]]` を基本にする

### 8.2 推奨ディレクトリ構成

```text
content/
  assets/
  samples/
  {任意名フォルダ}/
```

- `assets/` は画像や添付ファイルを置くが、ここ以外に置くことも許容する。
- `samples/` は parser 検証や表示サンプル用の原稿を置く。本番公開対象には含めない

### 8.3 推奨Frontmatter

```yaml
---
title: ページ表示名
aliases:
  - 別名1
  - 別名2
tags:
  - world
  - character
draft: false
created: 2026-03-13
updated: 2026-03-13
authors:
  - username
summary: 1行要約
---
```

- MVPでは構造化メタデータの正本は frontmatter とする
- MediaWiki風テンプレートから得られる同等情報は、取り込み時に frontmatter 相当の共通メタデータへ正規化する

### 8.4 編集記法の方針

同一ファイル内で Markdown 記法と MediaWiki 記法を混在させてよい。取り込み層が両方を認識し、共通の中間表現に正規化する。

#### 共通 (記法非依存)

- 内部リンク: `[[ページ名]]`, `[[ページ名|表示名]]`, `[[ページ名#見出し]]`
- テンプレート呼び出し: `{{テンプレート名|位置引数|名前=値}}`
- インライン太字: `**太字**` / `'''太字'''`
- インライン斜体: `*斜体*` / `''斜体''`
- インラインコード: `` `code` ``

#### Markdown / Obsidian 系

- 見出し: `## H2`, `### H3`, `#### H4`
- リスト: `- 項目`, `1. 項目`
- 埋め込み: `![[画像.png]]`
- callout: `> [!note] タイトル`
- 外部リンク: `[表示名](URL)`, `<URL>`
- frontmatter: YAML `---` ブロック

#### MediaWiki 系

- 見出し: `== H2 ==`, `=== H3 ===`, `==== H4 ====`
- リスト: `* 項目` (箇条書き), `# 項目` (番号付き)
- ネストリスト: `**` / `##` / `*#` (混合可。深さは記号の個数で決定)
- 定義リスト: `; 用語` (dt), `: 説明` (dd)
- テーブル: `{| ... |}` ブロック (`|-` 行区切り, `!` ヘッダセル, `|` データセル, `|+` キャプション, `rowspan`/`colspan`)
- 脚注: `<ref>注釈テキスト</ref>`, `<references />`
- 引用: `<blockquote>引用テキスト</blockquote>`
- 画像: `[[ファイル:画像名.png|オプション|キャプション]]` / `[[File:...]]`
- カテゴリ: `[[Category:カテゴリ名]]` (tags に変換)
- 改行: `<br>` / `<br/>`
- 水平線: `----`
- 主要記事リンク: `{{main|記事名}}`
- パース抑制: `<nowiki>リテラルテキスト</nowiki>`
- 詩文: `<poem>改行を保持するテキスト</poem>`
- コードブロック: `<syntaxhighlight>` / `<source>` (整形済みテキストとして表示)
- リダイレクト: `#REDIRECT [[ページ名]]` (ファイル先頭。別名として登録)
- マジックワード: `__TOC__`, `__NOTOC__`, `__FORCETOC__` (除去。目次制御は Phase 2)

### 8.5 内部リンク、バックリンク、未作成記事

- ビルド時に全原稿を走査し、内部リンクの解決結果を確定する。
- リンク先ページが存在する場合は通常の内部リンクとして出力する。
- `aliases` によって一意に解決できる場合は、そのページへの正規リンクまたはリダイレクトとして扱う。
- 同じ名称や alias が複数ページに対応する場合は、曖昧さ回避ページまたは同等の案内を生成できるようにする。
- リンク先ページが存在しない場合は未解決リンクとして記録し、公開サイトでは赤リンク相当の見た目で出力する。
- 各公開ページには、そのページを参照しているページ一覧をバックリンクとして表示できるようにする。
- 「XXはこちら」のような案内は、個別本文への手書きではなく、alias や曖昧さ回避のメタ情報から生成できることを優先する。
- 未解決リンクはビルド時エラーにはせず、設計上の「未作成ページ候補」として扱う。
- ただし、リンク切れの増加を把握するため、未解決リンク一覧はビルド成果物またはレポートとして出力できるようにする。

### 8.6 Wiki記法 / Markdown記法の互換方針

- 執筆記法は MediaWiki風のWiki記法を基礎とし、標準MarkdownとObsidian互換記法を同一原稿内で許容する。
- 最低限サポートする拡張は `[[ページ名]]`、`![[埋め込み]]`、`{{テンプレート名|...}}`、callout、frontmatter とする。
- 新しい意味付けが必要な場合は、特定見出し名や特殊な本文文字列のマジック解釈より、frontmatter や明示的なテンプレート記法を優先する。
- 入力記法の差異はビルド時の正規化層で吸収し、下流では見出し、段落、リンク、テンプレート呼び出し、メタデータなどの小さな共通表現へ寄せる。
- MediaWiki風テンプレートと実際の表示の変換規則は別レイヤに置き、テンプレートごとの差し替えや追加ができるようにする。
- ただし変換レイヤは公開用データ形式を肥大化させず、必要最小限の共通データモデルへ落とし込むことを原則とする。
- バックリンクや更新履歴の表示位置は、本文内の特定見出しで制御するのではなく、テーマやレイアウト側で決めることを基本とする。
- これにより、原稿は MediaWiki風の既存原稿、Markdown、Obsidian のいずれから来ても、公開時には共通の挙動へ寄せられるようにする。

### 8.7 ファイル名ルール

- MVPでは「参加者が自然に読めること」を優先し、日本語ファイル名を許容する。
- ただしObsidian公式の案内に合わせ、リンク解決を壊しやすい文字は避ける。
- URLの見た目が問題化した場合のみ、第2段階でslug運用を導入する。

### 8.8 原稿と公開物の分離

- 執筆者が編集する原稿は Wikiテキスト / Markdown 原稿を正本とする。
- 公開サイトで配信するページは、原稿から生成された HTML を正とする。
- 生成された公開ページは原稿とは別物として扱い、手編集しない。
- 公開可否の判断は原稿に対して行い、承認後のマージを契機に生成と公開を行う。
- したがって、公開フローは「原稿を直接見せる」のではなく、「原稿を変換した成果物を見せる」構成を前提にする。

## 9. 役割と運用フロー

### 9.1 想定ロール

- 読者: 公開サイトを閲覧する
- 執筆者: ページを追加・更新する
- 投稿者: GitHub Issue Formや外部フォームで記事データだけ送る
- 編集者: Pull Requestをレビューして整合性を保つ
- 管理者: マージ権限を持ち、公開可否を最終判断する

### 9.2 更新フロー

投稿フローは複数レーンを持つが、公開変更はすべてPull Requestへ集約する。
各レーンの詳細は [contribution-workflows.md](./contribution-workflows.md) を参照。

### 9.3 公開ルール

- `draft: true` のページは公開しない
- 画像や添付は公開前提のものだけを置く
- `content/samples/` 配下の原稿や、明示的にサンプル扱いされた原稿は本番公開対象に含めない
- 非公開メモや作業メモは公開用コンテンツと分離する
- すべての公開変更はPRを経由し、管理者マージを必須にする

### 9.4 原稿の取り込みフロー

1. 参加者が各自のローカル環境で原稿を編集する
2. 原稿を GitHub へ Pull Request として提出する
3. 管理者または編集者がレビューして、記事として取り込むかを判断する
4. 承認された原稿は `main` へマージする
5. マージを契機に静的サイト生成処理が原稿を解析・正規化し、HTML へ変換する
6. 生成済みサイトを GitHub Pages へ反映する
7. 読者が公開ページを閲覧できるようにする

### 9.5 未作成記事の公開時挙動

- 未作成記事への内部リンクは、通常リンクではなく赤リンク相当の見た目で表示する。
- 赤リンクを開いた場合は、単純な 404 ではなく「未作成記事ページ」または同等の案内ページを表示する。
- このページには少なくとも次を表示できるようにする。
  - 対象記事名
  - この記事が未作成であること
  - その記事名を参照している元ページ一覧
  - 近い既存ページ候補
  - 投稿や執筆への導線
- これにより、未作成記事はリンク切れではなく、将来作成される見込みのある概念として扱う。

### 9.6 ページ履歴と更新情報

- ページ単位の更新履歴は、原稿ファイルに対応する Git の履歴から生成できるようにする。
- 少なくとも各ページに最終更新日を表示し、可能であれば更新履歴一覧または GitHub 上の履歴への導線を持たせる。
- 詳細な差分閲覧は GitHub 側へ委譲してもよいが、読者が更新の存在を把握できる最小限の情報は公開サイト側にも持たせる。
- ビルド時に履歴を抽出する場合は、`git log --follow` 相当で改名追跡を考慮する。

### 9.7 管理者向け保守情報

- 孤立ページ、行き止まりページ、未解決リンク一覧、リンク数の偏りなどは、主に管理者向けの保守情報として扱う。
- これらは公開サイトの読者向けUIへ必須で載せるのではなく、管理アプリ、ビルドレポート、または Obsidian 上の補助情報へ寄せてもよい。
- したがって、公開側のMVP要件とは分離し、運用支援機能として別レイヤで扱う。

## 10. UI/UX要件

### 10.1 必須

- サイト内検索があること
- バックリンク一覧があること
- 未作成記事リンクを赤リンク相当で視認できること
- 赤リンク遷移時に未作成記事用の案内ページを表示できること
- タグまたはフォルダ単位で辿れること
- モバイルでも読めること
- 記事本文の書式 (太字・斜体・リスト・外部リンク・コード) が正しく表示されること
- テンプレート由来の構造データ (インフォボックス等) が閲覧用の定型表示に変換されること

### 10.2 推奨

- 最終更新日と更新履歴への導線を持つこと
- ページプレビューとパンくずを持つこと
- 記事内の見出しから自動生成された目次を表示すること
- カテゴリ名からそのカテゴリの記事一覧へ遷移できること

## 11. リスク

- Gitに不慣れな参加者がいる場合、運用教育が必要
- 同一ページの同時編集ではマージコンフリクトが起こる
- GitHub Freeで非公開リポジトリにしたい場合はプラン制約に当たる
- ブラウザ編集を後付けする場合、認証方式の再設計が必要
- GitHubアカウントなし投稿はスパム対策が必要
- 外部フォームを自動PR化する場合、別途バックエンドか外部サービスが必要

## 12. 将来拡張

- 記事ごとの静的 HTML 生成 (SPA から SSG への移行)
- ブラウザ内編集UIの追加
- PRごとのプレビュー環境
- テンプレート挿入や独自ショートコード
- talk page相当の議論導線
- 承認フローやLintの自動化
- 外部フォームからPRを自動生成する連携

## 13. 基本思想

- 最初に目指すべきは「Wikiサービスの完全代替」ではなく、「Wikiっぽく書けて静的に公開できる共同創作基盤」。
- そのため、MVPは公開面の完成度よりも執筆体験の自然さを優先する。
- 具体的には「Pages上で編集する」のではなく、「外部エディタで編集し、Pagesで読む」を基本思想にする。

## 14. 未決事項

### 14.1 運用方針

- 参加者は全員GitHubアカウントを持てるか
- 公開リポジトリで問題ないか
- 直接push可能なメンバーとPR必須メンバーを分けるか
- 画像アップロード頻度はどれくらいか
- ページ名に日本語を使う方針でURL体裁を許容できるか
- 投稿経路の選択は [contribution-workflows.md](./contribution-workflows.md) の未決事項を参照

### 14.2 技術方針

- どの MediaWiki風テンプレートをMVPで受けるか (少なくとも `基礎情報 国` は確定)
- 現在の SPA 構成をいつ記事ごとの静的 HTML 生成へ移行するか
- Markdown 本文の HTML レンダリングに外部パーサ (marked, markdown-it 等) を導入するか、自前で書くか
- 記事数が増えた場合の `content-data.js` 肥大化への対処 (分割ロード or 静的 HTML 化)
